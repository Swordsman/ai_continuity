NOCCO Protocol

Nuclear-Option Conversation Carryover

Version 1.0 | December 2025

What is NOCCO?

NOCCO is a "resurrection directive" - an instruction to create a document that captures a conversation's semantic meaning and progress so thoroughly that a different human with a different AI could continue the work with zero wasted effort.

Core principle: Preserve semantic meaning, not transcripts. Enable complete project resurrection even if original participants and context cease to exist.

When to Use NOCCO

  • Conversation approaching token limit (70-80%)
  • Major milestone completed
  • Before extended break from work
  • Need to switch between AI assistants
  • Context becoming complex or tangled

What to Include in a NOCCO

Essential elements:

  1. Context: Why this conversation exists, background information
  2. Decisions Made: What was decided and why (including alternatives considered)
  3. Work Completed: What's been done so far
  4. Current State: Where things stand right now (files, dependencies, status)
  5. Next Steps: What needs to happen next (immediate, short-term, long-term)
  6. Technical Details: Specifications, code snippets, configurations
  7. References: Links, documentation, key resources

Not needed: Complete transcripts, tangential discussions (unless they led to decisions), temporary debugging details

NOCCO Structure

No mandatory structure - use what serves resurrection. Common approach:

# NOCCO_##_[PROJECT]_##.md

## Executive Summary
[2-3 paragraph overview of current state]

## Context / Background
[Why this exists, what problem it solves]

## Work Completed
[What's been done since last NOCCO or from start]

## Current State
[Detailed snapshot: files, dependencies, what works/doesn't]

## Next Steps
[Immediate actions, short-term goals, long-term plan]

## Technical Details
[Specifications, decisions, configurations]

## Reference Material
[Key docs, links, quick reference tables]

## Meta Information
[Token count, how to continue, file locations]

Naming Convention

Format: NOCCO_##_[NAME]_##.md

Example: NOCCO_01_DASHBOARD_01.md

Why duplicate numbers? Both prefix and suffix ensure parent NOCCOs and corrections stay adjacent in any file sorting system.

Corrections:
  • Original: NOCCO_05_PROJECT_05.md
  • Correction: NOCCO_05_PROJECT_06.md

Keep first number (chain position), increment second (correction version).

Creating a NOCCO

How to request:

"Create a NOCCO for this [project/conversation]. Include everything needed for someone else to continue with zero wasted effort."

AI should:

  • Identify essential information for resurrection
  • Organize logically with clear sections
  • Be specific (file paths, versions, dates, exact decisions)
  • Verify: Would this alone be sufficient to continue?
  • Keep focused (semantic record, not transcript)

Using a NOCCO to Continue

Starting new conversation:

  1. Upload the NOCCO file
  2. Say: "Let's continue from this NOCCO"
  3. AI reads and confirms understanding
  4. Work continues from where it left off

AI should:

  • Read executive summary first
  • Check for any errata/corrections
  • Understand next steps clearly
  • Reference other sections as needed during work
  • Don't try to memorize everything - reference as needed

Key Principles

Philosophical Immutability

NOCCOs are philosophically immutable - conversations don't change after they happen, so their records shouldn't either. However, they can be:

  • Synthesized into larger documents
  • Combined and analyzed
  • Referenced and built upon
  • Edited for typos/minor fixes (substantive changes = create correction)

Corrections Travel Adjacent

Filename convention ensures corrections stay with originals. Even if some NOCCOs in a chain are lost, corrections for surviving ones remain intact.

Multi-AI Compatible

NOCCOs work with any AI: Claude, GPT, Gemini, DeepSeek, Perplexity, etc. Switch AIs by creating NOCCO at end of one conversation, uploading to next.

Not Just Projects

NOCCO works for conversations, research, discussions, creative work - anything where continuity matters.

Chains and Trees

Linear chain (most common):

NOCCO_01 → NOCCO_02 → NOCCO_03 → NOCCO_04

Branching (exploring alternatives):

NOCCO_01 → NOCCO_02 → NOCCO_03_optionA
                    → NOCCO_03_optionB

Merging (synthesis):

NOCCO_03_optionA ↘
                  → NOCCO_04 (combines insights)
NOCCO_03_optionB ↗

Best Practices

Do:
  • ✓ Be specific (file paths, versions, dates, exact decisions)
  • ✓ Focus on semantic meaning, not transcripts
  • ✓ Include everything needed for resurrection
  • ✓ Create proactively (don't wait for token limit)
  • ✓ Test: Could someone else continue from this alone?
Don't:
  • ✗ Be vague ("we decided on the approach" - which approach?)
  • ✗ Assume context ("obviously using Python" - state it explicitly)
  • ✗ Wait until 95% token limit
  • ✗ Include full transcripts or unedited discussions
  • ✗ Edit published NOCCOs for substantive changes (create correction instead)

Example Minimal NOCCO

# NOCCO_01_WEATHERAPP_01.md

## Context
Building weather dashboard aggregating 3 APIs. Started Dec 4, 2025.

## Decisions
- React + Node.js + PostgreSQL
- OpenWeather, WeatherAPI, Weatherstack APIs
- Recharts for visualization
- Redis caching (15min TTL)

## Work Completed
- API clients written (src/api/)
- Database schema implemented
- Basic UI components (Header, ForecastCard, SearchBar)

## Current State
- 8 files created, all in Git
- APIs tested, rate limiting works
- Ready for chart integration

## Next Steps
1. Integrate Recharts for forecast display
2. Add error handling UI
3. Deploy to Vercel

## Files
src/api/openweather.js, weatherapi.js, weatherstack.js
src/components/Header.jsx, ForecastCard.jsx, SearchBar.jsx
migrations/001_initial_schema.sql

Typical Length

  • Minimal: 1-3k tokens (simple conversations)
  • Standard: 5-15k tokens (typical projects)
  • Comprehensive: 20-40k tokens (complex multi-phase work)

Aim for "sufficient for resurrection" not "maximum detail possible".

The Resurrection Test

A NOCCO passes if, given only the NOCCO file, a competent person with an AI assistant could:

  • Understand what was being done
  • Understand why decisions were made
  • See the current state clearly
  • Continue the work seamlessly
  • Not redo anything already done

Quick Start

  1. Work with AI on project
  2. When approaching natural breakpoint: "Create a NOCCO"
  3. AI generates comprehensive snapshot
  4. Download and save the NOCCO
  5. When continuing (new conversation): Upload NOCCO, say "Continue from this"
  6. AI reads and continues seamlessly
